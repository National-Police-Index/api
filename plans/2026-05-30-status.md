## Current Status

### Repo layout
- `resolve/` — canonical pipeline (formerly `resolve_seq/`)
- `resolve_archive/match_old.py` — the older `match.py`, superseded by `match_fast.py`
- `resolve_archive/resolve_batch/` — batch-optimization experiments, kept for reference
- `resolve_archive/resolve_old_untracked/` — empty old `resolve/` (just a debug log)

`match_fast.py` is the canonical pipeline — it produces every file currently in `data/output/` (`auto_matched.jsonl`, `early_filtered.jsonl`, `failed_entity_resolution.jsonl`, `df_fast.csv`, `unmatched_fast.xlsx`). The older `match.py` was identified as stale by output-file pattern and archived.

The batch pipeline was archived because (a) it produced different match sets from the sequential pipeline that were never reconciled, and (b) its batch county lookup was broken (only resolved 1–2 of 45 agencies due to a Supabase 1000-row cap). See `resolve_archive/resolve_batch/COUNTY_LOOKUP_TODO.md` if we ever revisit.

### Groundtruth tests against `resolve/data/output/auto_matched.jsonl` (9875 auto-matched officers)

`tests.py` — **100.0% pass rate (2/9875 failed)**
- `agency_name_similarity`: 2 failures — both legitimate concerns (DA→Hayward PD), not test bugs. These motivated the fix below.
- All other tests clean.

`test_names.py` — **0 flagged** (was 392; LLM said 325 NEEDS_REVIEW)
- Old test over-flagged by treating "another similar name exists in POST" as ambiguity even when the matched POST first name was an exact match to input.
- Fixed: only flags when matched first name is non-exact AND another POST person is at least as close.
- All 9875 current auto-matches have exact first-name equality, so 0 ambiguity in this dataset.

### DA→PD auto-match leak — guard added, not yet verified end-to-end

`helpers.validate_agency_match` now hard-rejects when every input agency is non-LE (DA / Coroner / ME / Public Defender / AG) and the POST agency is LE (Police / Sheriff / Marshal / Patrol / Probation / Corrections / Public Safety). The LLM is bypassed for this case — the LLM had occasionally returned MATCH for "Alameda County DA" → "Hayward PD", producing the two known-bad rows `A19-I99` (KENNETH LANDRETH) and `B04-J30` (SCOTT LUNGER), both already in the README's Remove list.

Guard is bypassed when any LE agency appears in `mentioned_agencies` (e.g. "Public Defender" source + "Oakland PD" in mentioned_agencies still validates). Verified with 5 unit-style cases.

**Not verified**: the guard hasn't been exercised against a full pipeline run because that needs the API server up.

## Next Steps

### 1. Verify the DA→PD guard end-to-end
Start the API server (`cd server && python3 src.py`) and re-run `match_fast.py`. Confirm:
- A19-I99 and B04-J30 no longer appear in the new `auto_matched.jsonl`.
- They DO appear in `failed_entity_resolution.jsonl` with the new rejection reason.
- `tests.py` pass rate stays at 100% (or goes to 100.0% from 99.98%).
- No regression: total auto-match count drops by at most a handful (we're only blocking a narrow non-LE-only pattern).

### 2. Sweep the README "Remove" list against the new run
The README has ~50 POST IDs the user has flagged as bad matches. After step 1, check how many are now auto-removed by the guard vs. how many still leak through. The ones that still leak are the next bug patterns to characterize.

### 3. Address the three open to-dos at the bottom of `resolve/src/README.md`
- Fuzzy agency-name matching for county lookup (so "Sheriff's Office" / "Sheriff's Department" / "/Coroner" variants resolve to the same county — exactly the normalization already in `tests.py`).
- Generate candidates for common last names instead of dropping them; route to manual review.
- Regenerate `mentioned_agencies` with updated logic (referenced by commit `7c201df`).

### 4. Lift the test-side agency normalization into the pipeline
`tests.py:_normalize_agency` and `_agency_token_similarity` are now the most accurate non-LLM agency-equivalence check in the codebase. The pipeline currently relies on `helpers.validate_agency_match` which calls an LLM. A normalized token-set pre-check could short-circuit obvious matches and cut LLM calls substantially. Worth measuring before committing to.

### 5. Treat `auto_matched.jsonl` as a regression fixture
After any matching-logic change, re-run `tests.py` and `test_names.py`. Both run in seconds on clean data (`test_names.py` only calls the LLM when its post-filter actually flags something, which is currently zero).
